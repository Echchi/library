> 아무리 복잡한 어플리케이션이라 해도 데이터를 입력받아 처리하고 그 결과를 출력하는 것이 전부다
## 1. 변수
### A. Summary
- 변수가 필요한 이유?
	- 기억하고 싶은 값을 메모리에 저장하고 저장된 값을 읽어들여 재사용하기 위해
- 변수
	- 하나의 값을 저장하기 위해 확보한 메모리 공간 자체 또는 그 메모리 공간을 식별하기 위해 붙인 이름
	- 할당 : 변수에 값을 저장하는 것
	- 참조 : 변수에 저장된 값을 읽어 들이는 것
### B. Rationale
- 자바스크립트는 값을 저장하는 것이 아닌 값이 저장된 메모리 주소를 가리키는 식별자를 저장한다
### C.  Application
- React에서 state는 값이 아니라 **state 변경을 참조(reference) 비교로 감지**할 수 있음
- useCallback, 혹은 useMemo, React.memo 를 사용하여 주소값을 메모이제이션하면 불필요한 재렌더링을 막을 수 있는 이유도 바로 그 주소값을 저장해두는 것이기 때문

## 2. 변수 선언
### A. Summary
- 값을 저장하기 위한 메모리 공간을 확보하고 변수 이름과 확보된 메모리 공간의 주소를 연결해서 값을 저장할 수 있게 준비하는 것
### B. Rationale
- 변수 선언은 평가 단계에서 처리되며 var는 **선언과 동시에 초기화까지 이루어져 undefined를 바인딩**함 
- 이는 실행 중 참조 시 “식별자를 찾지 못하는 에러”를 방지하는 목적이 있음

### C. Application
- React에서 렌더링 시점과 변수 초기화 시점이 어긋나면 초기화되지 않은 변수를 참조해 `undefined`가 나오는 경우가 있다.

## 3. 호이스팅
- 소스코드의 평가 과정에서 자바스크립트 엔진이 변수 선언을 포함한 모든 선언문을 소스코드에서 찾아내 먼저 실행하는 것
- 변수 선언문이 코드의 선두로 끌어 올려진 것처럼 동작하는 자바스크립트 고유의 특징

## 4. 값의 할당
- 변수 선언은 소스코드가 순차적으로 실행되는 시점인 런타임 이전에 먼저 실행되지만 값의 할당은 소스코드가 순차적으로 실행되는 시점인 런타임에 실행

## 5. 값의 재할당
### A. Summary
- 상수 : 값을 재할당할 수 없어서 변수에 저장된 값을 변경할 수 없는 값
### B. Rationale
- 값을 수정하는 것이 아니라 새로운 메모리 주소를 확보 후 값을 연결하는 것
- 쓰지않는 메모리 값에 대한 정리가 필요
### C. Application
- React에서 state와 props는 **값이라기보다 참조(레퍼런스)** 관점으로 다루어진다.  
- React.memo는 props의 **참조가 바뀌었는지**(얕은 비교)를 기준으로 리렌더링 여부를 결정한다.
